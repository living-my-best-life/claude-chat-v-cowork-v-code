1. What Cowork Actually Is

Cowork is not a chat interface with a new skin. It's an autonomous desktop agent built into the Claude Desktop app.

When you open the Cowork tab, you're giving Claude access to a sandboxed Linux VM on your machine. Inside that sandbox, Claude can write code, execute scripts, create files — real Word docs, slide decks, spreadsheets, PDFs — and connect to services like Gmail, HubSpot, Slack, and Google Drive. You don't configure the infrastructure. Anthropic manages it.

You describe what you need. Cowork plans the work, breaks it into sub-agents that run in parallel, and delivers output as clickable files you can open directly.

GTM Example

"Build a competitive battlecard for Gong vs. our platform. Pull from their recent blog posts, G2 reviews, and their latest pricing page. Output as a Word doc I can share with the sales team."

Cowork plans the research, spawns parallel sub-agents for each source, synthesizes findings, and drops a formatted .docx into your folder. Not an artifact in a chat bubble — a real file you can edit in Word.

Plans & Tracks Work

Decomposes complex GTM tasks into subtasks. You watch progress in real time and steer mid-task.

Parallel Sub-Agents

Spawns independent Claude instances that work on different parts simultaneously. Research + writing + formatting at once.

Real Files

Creates actual .docx, .pptx, .xlsx, and .pdf files — delivered to your folder, ready to edit and share.

Connected to Your Stack

Gmail, GitHub, Slack, Google Drive via built-in connectors. Plus any tool via MCP servers.

Keep in Mind

Cowork runs sandboxed — it can't touch your OS or files outside the folder you shared. But inside that folder, it has full read/write/delete access. Grant folder access intentionally.

2. Cowork vs. Chat: Why It's a Different Beast

You're already using Claude Chat in the Desktop app. So what does Cowork add? Here's the honest breakdown.

Capability Chat Cowork
Task decomposition Manual only Automatic planning
Sub-agent coordination None Parallel sub-agents
File output Artifacts (copy/paste) Real files to your folder
Code execution None Full sandboxed Linux VM
Data analysis Limited (text-based) Python, pandas, charts
Connectors (Gmail, Slack…) Available Available + more powerful
Plugins & skills Skills only (via settings) Full plugin panel
Progress tracking None Visual task list
Multi-step workflows Single conversation Orchestrated workflows

The short version: Chat is for conversations. Cowork is for workflows.

If you're asking a question, refining copy, or brainstorming positioning — Chat is fine. If you're building a QBR deck, analyzing six months of pipeline data, or prepping competitive intelligence across multiple sources — that's Cowork territory.

GTM Leader Take

The pattern I see across every GTM team I've advised: they start in Chat, hit a ceiling, and wonder why AI "isn't working." Cowork removes that ceiling. It's the difference between having a smart assistant you text and having one who sits at a desk and works.

3. Plugins and Skills for GTM

When Anthropic unveiled AI tools automating legal and financial research in early 2026, legacy software stocks dropped $285 billion in a single day. Investors saw AI agents moving into the application layer — and repriced the entire software sector.

The plugins in your Cowork sidebar are part of what triggered that reaction. Here's what matters for GTM leaders.

What Are Skills?

Skills are reusable instruction manuals that teach Claude how to approach specific, repeatable tasks. Say "create a PowerPoint" and the pptx skill loads. Say "analyze this CSV" and the data exploration skill kicks in. You can also trigger skills explicitly — type / in Cowork for autocomplete.

Skills don't all load at once. Claude reads a short description of each (~100 tokens) to decide relevance, then loads full instructions only when needed. Your context window stays clean.

4. MCPs: Connecting Cowork to Your GTM Stack

MCP stands for Model Context Protocol — the open standard by Anthropic. Each MCP server exposes tools Claude can call. A Gmail MCP gives Claude search_emails, send_email, read_email. A HubSpot MCP gives it search_contacts, get_deal, update_deal_stage. You get the picture.

There are three ways to connect MCP servers to Claude, and understanding the difference matters for your GTM stack.

Managed
Remote Connectors
One-click setup via Anthropic. Works everywhere incl. claude.ai. Gmail, Slack, etc.
Packaged
Extensions
Local MCP servers packaged for one-click install. Manage in Settings → Extensions.
Custom
JSON Config
Edit config file directly for CRM, outreach tools, custom internal APIs.

GTM Stack Connections That Matter

The connectors that change daily GTM workflows:

Email (Gmail / Outlook)

Search, read, draft, and send. Review inbound before meetings. Draft follow-ups from call notes.

Slack

Search channels, read threads, draft messages. Monitor deal rooms and competitive channels.

Google Drive

Search and read docs. Pull from competitive folders, battlecards, and enablement libraries.

CRM (via custom MCP)

Search contacts, pull deal data, analyze pipeline. Requires JSON config setup.

Per-Tool Permissions

This is the part most GTM leaders miss. For every connector, you can set individual tools to Allow (runs automatically), Ask (confirms before running), or Block (never runs).

Translation: you can let Claude search your email automatically but require approval before sending. Let it read Slack but block it from posting. This is how you build trust incrementally — and it's how I recommend every GTM leader start.

6. A 1-Minute Hack That Makes Claude Desktop 2x More Powerful

I genuinely can't believe more people in GTM don't know about Desktop Commander. It's the highest-ROI move you can make. Takes less than 60 seconds.

  1. Open Claude Desktop
  2. In a chat window click: "+ → Connectors → Manage connectors"
  3. Click: "Browse connectors → All → Desktop Commander"
  4. Select which tools don't require your approval

The result: Chat, Cowork, and Code Tab can do virtually anything on your laptop — including installing MCP servers or accessing any file.

GTM Use Case Example
"Go to the G2 page for [competitor], pull the top 10 negative reviews from the last 6 months, and build me a battlecard in Word format."
  • Claude navigates the web, extracts review data
  • Analyzes sentiment and categorizes objections
  • Creates a formatted .docx battlecard with objection-by-objection talk tracks
  • Drops it directly into your shared folder
Pro Tips

Disable the Claude in Chrome extension when not needed — Claude sometimes defaults to web-based actions even when it has a dedicated MCP connector. Toggle it off when doing local work.

Start with "Ask" permission for everything, then selectively move to "Allow" as you build confidence. None of these tools can take actions on their own unprompted.

7. How to Give Claude Cowork Cross-Session Memory

This is the one that changes everything for GTM leaders. By default, every Cowork session starts fresh. It doesn't remember your ICP, your competitive landscape, your deal stages, or your naming conventions. Two simple steps fix this.

  1. Enable Desktop Commander extension in "Settings → Connectors"
  2. Copy the prompt below to "Settings → Cowork → Global instructions"
## Memory Management When you discover something valuable for future sessions — competitive intel, deal context, ICP definitions, messaging frameworks, account notes — immediately append it to {your_folder}/memory.md Don't wait to be asked. Don't wait for session end. Keep entries short: date, what, why. Read this file at the start of every session.

This costs almost zero tokens and survives crashes, compaction, and new sessions.

Advanced: Structured Memory for GTM

For revenue leaders managing complex deal cycles, split into multiple files so Claude loads only what's relevant:

## GTM Memory System Maintain a structured memory rooted at .claude/memory/ ### Structure - memory.md — index of all memory files - general.md — ICP definitions, team structure, tool preferences - competitors/{name}.md — per-competitor intel and positioning - accounts/{name}.md — key account context and history - playbooks/{type}.md — messaging frameworks, objection handling - metrics.md — KPIs, targets, historical benchmarks ### Rules 1. When you learn something worth remembering, write it immediately 2. Keep memory.md as a current index with one-line descriptions 3. Entries: date, what, why — nothing more 4. Read memory.md at session start. Load others only when relevant 5. If a file doesn't exist yet, create it ### Maintenance When I say "reorganize memory": 1. Read all memory files 2. Remove duplicates and outdated entries 3. Merge related entries 4. Update memory.md index 5. Show me a summary of what changed
